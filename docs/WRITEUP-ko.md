# fixbench 회고 — NAVER 2026 하계 인턴 프로덕트 트랙 지원 동기

## 무엇을 만들었나

**fixbench**는 AI 코딩 에이전트를 *공정하게* 비교하기 위한 작은 평가 하네스입니다.
버그가 있는 저장소를 시도(trial)마다 임시 폴더로 격리해 복사하고, 교체 가능한
**에이전트**가 그 안에서 수정을 시도한 뒤, 두 가지 방식의 **채점기**로 결과를 매깁니다.

- **에이전트(플러그형)**: `noop`(아무것도 안 함, 하한 기준선) · `oracle`(정답
  `.solution` 적용, 상한 기준선) · `self-loop`(Anthropic SDK 기반 최소 tool-use 루프) ·
  `claude-code`(실제 Claude Code CLI를 headless로 호출하는 얇은 래퍼).
- **채점기(플러그형)**: `deterministic`(실제 테스트를 실행 — 타깃 테스트 통과 **그리고**
  가드 테스트 무회귀일 때만 성공) · `llm-judge`(LLM이 수정 코드를 *읽고* 판단해
  `passed / score / grounded / hallucinationRisk / rationale` 구조화 판정을 반환,
  실행하지 않음).
- 측정값: pass@1, pass@k, 회귀(regression) 횟수, 토큰, 비용, 지연시간, tool 호출 수.
- 명령어: `run`(단일 실행) · `compare`(에이전트 전체 순위 → `leaderboard.md` + `compare.json`) ·
  `calibrate`(심판 vs 결정론 혼동행렬) · `fix`(자동 수정 봇 → 패치 diff). 거기에 자동수정
  GitHub Action(통과 시 **드래프트 PR** 생성, 자동 머지 없음)과 리더보드를 띄우는
  Cloudflare Workers(`worker/index.ts`)까지 붙여 한 바퀴를 완성했습니다.

핵심 설계는 **에이전트와 채점기를 같은 인터페이스 뒤에 두는 것**입니다. 모든 후보가
동일한 격리 저장소·동일한 지표 파이프라인을 통과하므로, 측정에서 변하는 것은 *비교
대상 그 자체뿐*입니다. 이게 평가 하네스를 만드는 이유라고 생각했습니다.

## 왜 이것이 Tech②와 Tech③에 연결되는가

**Tech② (AI 코딩 에이전트 평가 하네스 + 자동 수정 봇)** — 하네스 본체와 자동수정 봇이
그대로 대응합니다. 격리(`isolate.ts`) → 에이전트가 수정 → `deterministic` 채점기가
테스트를 실행 → `compare`로 순위·리더보드 집계. "테스트를 통과시켜라"는 목표가 쉽게
악용된다는 점을 알기에, 타깃 테스트 옆에 **가드 테스트**를 두어 옆 기능을 깨뜨리는 수정을
회귀로 잡아냅니다. 그리고 **자동 수정 봇**(`src/fix.ts`)을 실제로 구현했습니다 — 격리된
사본에서 에이전트를 돌려 `proposed-fix.diff`(allowedFiles의 unified diff)와
`fix-result.json`을 내보내고, 원본 픽스처는 절대 건드리지 않습니다. 이걸 GitHub
Action(`.github/workflows/fixbench-autofix.yml`)이 감싸서 **결정론 채점을 통과한 경우에만
드래프트 PR**을 엽니다 — 자동 머지는 절대 하지 않고, 사람이 검토·머지합니다. "봇의 권한은
'제안'에서 끝난다"는 것이 자동수정 봇의 핵심 안전 자세라고 봤고, 채점은 모델의 *주장*이
아니라 *실행*으로 판정한다는 결정론 원칙을 끝까지 지켰습니다.

**Tech③ (RAG / 대화 평가 · LLM-as-a-judge)** — `llm-judge` 채점기와 보정(calibration)이
대응합니다. 정답을 실행할 오라클이 없는 상황(예: RAG 답변, 대화 응답)에서 모델 심판이
유일하게 확장 가능한 신호라는 점에 착안했습니다. 판정은 `tool_choice`로 단일
`submit_verdict` 툴 호출을 강제해 **구조화 출력**으로 받고, 프롬프트는 "근거 없으면
PASS보다 FALSE"를 명시해 자신 있는 거짓 통과를 억제했습니다. 그리고 결정론 채점기를
*지상 진실(ground truth)*로 두어 **`src/calibrate.ts`**로 심판의 오류율을 실제로
정량화합니다 — agreement·precision·recall과 false-pass/false-fail 혼동행렬까지. 합성
데이터 self-test에서 agreement는 **70%**, false-pass가 **2건** 나오는데(자동수정 봇이
심판만 믿으면 깨진 패치를 통과시킬 수 있는 위험한 방향), 바로 이 숫자가 "심판은 신뢰하기
전에 측정해야 한다"는 Tech③의 핵심을 손에 잡히게 만든다고 생각합니다.

## 솔직한 한계와 학습 의지

과장 없이 적습니다. 이 프로젝트가 보여주는 것은 "사고방식"이지 완성된 제품이 아닙니다.

- `tasks/`는 제가 직접 큐레이션한 5개짜리 **마이크로 벤치마크**이며 SWE-bench가
  아닙니다. `self-loop`은 의도적으로 *최소* 구현입니다. `config.ts`의 가격은 **미검증**
  플레이스홀더라 비용은 근사치입니다(토큰 수는 API에서 받은 정확값). `calibrate
  --self-test`의 70% 수치는 **합성 데이터**라 방법론을 보여줄 뿐 실제 심판 정확도가
  아닙니다 — 제대로 된 특성화에는 수십~수백 건의 라벨이 필요합니다. `run`/`compare`/
  `calibrate`/`fix`·드래프트 PR Action·Cloudflare Worker는 모두 구현·검증을 마쳤고
  (`wrangler deploy --dry-run` 빌드 통과), 남은 건 *기능*이 아니라 *내 자격증명*입니다:
  LLM 실행에는 `ANTHROPIC_API_KEY`, 실제 `cf:deploy`에는 Cloudflare 로그인, Action
  활성화에는 GitHub 푸시 + repo secret이 필요합니다 — 코드에 있는 것만 사실로 적었습니다.

- **Tech②의 부족분**: 저는 프런트엔드 출신이라 Kubernetes 운영과 Python 깊이는 아직
  학습 영역입니다. 다만 "격리·재현·결정론 채점·회귀 가드·기준선으로 상하한 묶기" 같은
  평가 시스템의 사고는 언어와 무관하게 옮겨갈 수 있다고 믿고, 이번 하네스로 그 사고를
  실제 코드로 입증하려 했습니다. 인턴 기간에 k8s·Python 스택을 빠르게 따라잡겠습니다.

- **Tech③의 부족분**: 본격적인 RAG 파이프라인(청킹·임베딩·리트리벌 튜닝) 실무 경험은
  아직 없습니다. 대신 LLM-as-a-judge의 핵심인 *구조화 출력 강제*와 *지상 진실 대비
  보정*을 직접 구현해, 평가자 자체를 신뢰하기 전에 측정해야 한다는 원칙을 손으로
  익혔습니다. 검색·근거(grounding) 평가로 자연스럽게 확장 가능한 토대라고 생각합니다.

## 마무리

좋은 평가 하네스의 가치는 화려함이 아니라 **공정함과 오류에 대한 정직함**에서 나온다고
생각합니다. fixbench는 작지만 그 두 가지 — 모든 후보를 같은 파이프라인에 세우는 공정함과,
무엇이 한계인지(미검증 가격·마이크로 벤치마크·합성 self-test 수치)를 코드와 문서에 분명히
적는 정직함 — 을 보여주려 했습니다. 결정론 채점·자동수정 봇(드래프트 PR)·LLM 심판·보정까지
한 바퀴를 실제로 돌려봤고, 부족한 스택은 배우겠다는 의지로 채우겠습니다.
